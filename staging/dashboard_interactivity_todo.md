## Status: Phase 3 ships visual-only interactivity

Current behaviour in `docs/dashboard.html` Mission Control section:

- `<details>/<summary>` collapse-and-expand for "待你決策" decision cards — **works client-side only**
- `<form onsubmit="return false">` with `<button>` labelled "我知道了" / "稍後再看" — **purely visual dismiss** (closes the `<details>` pane in the user's current tab, no state persisted)
- No real POST endpoint exists yet
- Reloading the page re-shows the same cards because nothing is written back

## Phase 4 requirements (to be built)

### 4.1 Dismiss action → server write-back
- User clicks "我知道了" on a decision card
- Browser POSTs `{event_ts, action: "dismissed", ts_client}` to a lightweight endpoint
- Endpoint appends an `agent_progress.jsonl` event `{type: "decision_dismissed", status: "done", msg: "Edward 已看過", actor: "edward"}`
- Next dashboard rebuild filters out dismissed cards

### 4.2 Approval action → ratification
- "稍後再看" on an SOP proposal card flips to "批准" and "退回" two buttons
- "批准" writes `staging/approvals/<sop_id>_approved.md` + progress event
- Daemon picks up approval file and advances the SOP through the publish pipeline

### 4.3 Backend options (lowest friction first)
1. GitHub Actions `repository_dispatch` webhook — zero infra, ~30s latency, free
2. Fly.io micro-service — persistent, <1s latency, ~$0/mo on free tier
3. Local Python HTTP server via Cloudflare Tunnel — only works when laptop is on

**Preferred**: Option 1 (GitHub Actions dispatch) — matches the existing "GitHub Actions chained" migration decision.

### 4.4 Auth model
- Only Edward should be able to dismiss/approve
- Use a per-dashboard HMAC token embedded in the HTML at render time
- Rotate token daily via build_dashboard.py

### 4.5 Progress feed streaming
- Current dashboard auto-refreshes every 5 minutes via meta refresh
- Phase 4: optional EventSource (SSE) poll of progress jsonl for near-real-time feed
- Fallback gracefully to meta refresh if SSE endpoint unreachable

## Out of scope for Phase 4

- Editing past events (immutable append-only log)
- Multi-user approval workflow (Edward is sole approver)
- Rich text / attachments in decision cards
