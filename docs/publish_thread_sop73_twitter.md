# Twitter Thread — SOP #73: Dynamic Tree Protocol

**Hook (tweet 1):**
You have 6 active projects.

All valid. All need work.

You pick the one you last thought about.

3 months later: 2 projects dead from neglect. 1 over-built. 0 at target.

That's not prioritization. That's entropy.

SOP #73: The Dynamic Tree Protocol — 4-gate branch prioritization.

---

**Tweet 2:**
The core mistake: treating your project list as a to-do list.

A to-do list is flat. You check things off.

A dynamic tree is a **derivative calculator**.

Each branch has a current rate of change toward your core goal.

Your job: route effort to the highest derivative.

---

**Tweet 3:**
G0 — Map all active branches.

For each one:
- When was it last touched?
- What's its current milestone?
- What changes if you push it 1 cycle?

Most people can't answer the third question.

That's the gap.

---

**Tweet 4:**
G1 — Calculate derivatives.

Four derivative types:

→ **Revenue derivative**: moves toward economic self-sufficiency
→ **Capability derivative**: expands what you can do
→ **Coverage derivative**: closes a system gap
→ **Compounding derivative**: unlocks other branches

Rank by these. Not by recency. Not by discomfort.

---

**Tweet 5:**
G2 — Check regime.

Branch priority isn't static. It's regime-dependent.

Revenue deadline approaching? → Economic branches first
Behavioral gap found? → Calibration branches first
All branches healthy? → **Least-recently-touched branch first**

Read the regime before choosing the branch.

---

**Tweet 6:**
The least-recent rule:

When derivatives are within 20% of each other — push the branch you touched least recently.

Why: branches decay exponentially when untouched.

A branch you haven't pushed in 15 cycles has lost its context. Reconstructing context has a cost.

Don't let branches die from neglect.

---

**Tweet 7:**
G3 — Execute, then persist.

Not just: do the work.

But: write what changed → update branch status → run L2 verdict → if quality drops, trigger L3 evolution.

Output that doesn't persist doesn't count.

Cold start tomorrow, and that branch might as well be dead.

---

**Tweet 8:**
G4 — Recalculate next cycle.

The tree is dynamic.

What had the highest derivative last cycle may have the lowest today — because you moved it.

Anti-pattern: pushing the same branch 5 cycles because "it still has work to do."

That's comfort zone. Not optimization.

---

**Tweet 9:**
The Dynamic Tree Protocol isn't productivity hacks.

It's a **mathematical commitment**: every cycle, recalculate. Every cycle, route to highest derivative.

Stop recalculating = branches diverge from core goal.

The tree is alive only if you keep running the protocol.

---

**Tweet 10 (CTA):**
SOP #73: Dynamic Tree Protocol

G0: Map all branches
G1: Calculate derivatives (revenue / capability / coverage / compounding)
G2: Check regime
G3: Execute + persist
G4: Recalculate

The protocol is simple. The discipline is the hard part.
